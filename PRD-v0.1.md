## 一、产品概述

### 1.1 产品定位
一款面向**情侣和朋友**的微信小程序，通过**双人PK挑战**的方式，让减肥变得有趣、有动力、能坚持。

### 1.2 核心价值主张
- **不是一个人战斗** — 情侣/朋友相互监督，告别孤军奋战
- **有输赢才有趣** — 设定目标和奖惩，增加博弈感
- **轻量易分享** — 小程序即开即用，一键分享给好友

### 1.3 目标用户
| 用户画像 | 特征 | 痛点 |
|---------|------|------|
| **情侣用户** | 25-35岁，同居或异地 | 想找共同话题，一起变好 |
| **闺蜜死党** | 女性为主，25-30岁 | 经常喊减肥但坚持不下去 |
| **夫妻档** | 30-40岁，已婚 | 婚后发福，想找回状态 |

## 二、核心功能模块

### 2.1 MVP功能（第一版必须做）

#### 功能1: 创建挑战
- 发起者设定挑战参数：
  - 挑战时长（7天/14天/30天）
  - 减重目标（如：减重3kg）
  - 奖惩措施（可选：发红包、请吃饭、承包家务等）
- 邀请好友加入（微信分享卡片）
- 双方确认后挑战开始

#### 功能2: 每日打卡
- **体重记录**: 手动输入或拍照识别体重秤
- **饮食记录**: 拍照上传+简单描述（早/中/晚）
- **运动记录**: 选择运动类型+时长（可选关联微信运动步数）
- **打卡提醒**: 定时推送提醒双方打卡

#### 功能3: 双人PK对比（仅两人可见）
- **私密对决**: 只有挑战双方能看到数据，不是公开排行榜
- **进度对比**: 
  - 体重变化曲线对比图（两条线对比）
  - 减重百分比进度条（你 vs TA）
  - 打卡天数对比（连续打卡天数PK）
- **领先提示**: "你已领先XX 1.2kg，继续保持！"
- **胜负悬念**: 实时显示差距，比如"还差0.5kg就反超了！"

#### 功能4: 奖惩结算
- 挑战结束自动判定胜负
- 输方执行惩罚（系统提醒，线下执行）
- 胜方获得成就勋章
- 双方可选择继续新挑战或解散

#### 功能5: 社交分享
- **每日打卡分享**: 生成精美海报分享朋友圈
- **挑战结果分享**: 胜败结果海报，带有趣文案
- **邀请好友**: 一键分享给微信好友

## 三、用户流程

```
【发起者视角】
打开小程序 → 点击"发起挑战" → 设定参数 → 微信邀请好友 
→ 等待好友确认 → 挑战开始 → 每日打卡 → 查看PK榜 
→ 挑战结束 → 分享结果

【参与者视角】
收到邀请卡片 → 点击进入 → 查看挑战详情 → 确认加入 
→ 挑战开始 → 每日打卡 → 查看PK榜 → 挑战结束
```

## 四、技术方案（概要）

### 4.1 技术栈选择
- **前端**: 微信小程序原生框架（WXML + WXSS + JS）
- **后端**: Node.js + Express / 或云开发（微信云开发）
- **数据库**: MongoDB / 或微信云数据库
- **存储**: 微信云存储（图片）
- **部署**: 微信小程序云开发（低成本、免运维）

### 4.2 为什么选择微信云开发？
| 优势 | 说明 |
|------|------|
| 零服务器成本 | 按量付费，初期几乎免费 |
| 免运维 | 不用买服务器、配环境 |
| 快速开发 | 前后端一体化，开发速度快 |
| 微信生态集成 | 登录、分享、支付无缝对接 |

### 4.3 开发周期估算
- **MVP版本**: 4-6周（1个全栈开发）
- **开发人力**: 可外包或找兼职开发（预算1-2万）

## 六、关键设计原则

### 6.1 轻量化
- 不做复杂的社区功能
- 不做课程、食谱等内容
- 专注做好"双人PK"这一个核心场景

### 6.2 游戏化
- 进度可视化（进度条、曲线图）
- 即时反馈（打卡成功动画）
- 胜负悬念（实时显示差距）

### 6.3 情感化
- 情侣专属文案（"亲爱的，今天该打卡啦"）
- 趣味惩罚模版（"输了请喝奶茶"、"承包一周家务"）
- 分享海报带情感化文案

## 七、风险与应对

| 风险 | 应对措施 |
|------|----------|
| 用户打卡流失 | 增加提醒功能、设计连续打卡奖励 |
| 作弊问题 | 要求拍照打卡、AI识别体重秤 |
| 情侣分手后不用 | 扩展闺蜜/家人场景，不只限于情侣 |
| 微信审核不通过 | 提前阅读审核规范，避免敏感功能 |
